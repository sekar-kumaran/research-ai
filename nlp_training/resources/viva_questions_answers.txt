NLP Research Assistant Viva: 56 Questions with Answers

Q1. What is the main goal of your project?
A1. The goal is to build an AI-powered research assistant that can classify arXiv papers, retrieve semantically similar papers, summarize research text, and answer questions using retrieval-augmented generation (RAG).

Q2. What dataset did you use?
A2. I used arXiv parquet data, mainly from arxiv_part_A.parquet and arxiv_part_B.parquet, with optional support for additional chunks.

Q3. Why did you choose a pipeline-based architecture?
A3. A pipeline makes the workflow reproducible, modular, and scalable. Each stage has a clear responsibility: cleaning, EDA, feature engineering, training, and retrieval indexing.

Q4. Which script orchestrates the full flow?
A4. run_pipeline.py orchestrates all stages in sequence and passes useful runtime arguments like chunk size and fast mode.

Q5. Why did you process data in chunks?
A5. Chunking is memory-safe for large datasets. It prevents RAM overflow and allows processing millions of rows on moderate hardware.

Q6. What does NLP preprocessing include in your project?
A6. It includes lowercasing, URL removal, punctuation cleaning, token filtering, stopword removal, and lemmatization when spaCy is enabled.

Q7. Why is there a fallback from spaCy to NLTK-style processing?
A7. It improves robustness. If spaCy model loading fails, preprocessing still works using a lightweight fallback path.

Q8. How did you ensure deterministic data splits?
A8. I used hash-based bucketing with stable MD5 hashing on IDs, so the same document always goes to the same split across reruns.

Q9. Why is deterministic splitting important?
A9. It prevents train-test leakage drift when new data is added and supports fair model comparison over time.

Q10. What is the purpose of deduplication?
A10. Deduplication avoids repeated papers in training and evaluation, reducing bias and preventing inflated performance.

Q11. Why did you use SQLite for deduplication?
A11. SQLite allows disk-based uniqueness checks using INSERT OR IGNORE, which is memory-efficient compared to huge in-memory sets.

Q12. Explain this loop: for p in parts_dir.glob("part_*.parquet"):
A12. It iterates over all files in parts_dir whose names match the pattern part_*.parquet, usually old partial outputs from previous runs.

Q13. What does glob("part_*.parquet") do?
A13. glob performs filename pattern matching and returns all matching path objects in that directory.

Q14. What does p.unlink() do?
A14. unlink deletes that file path from disk. Here it clears previous part files for a fresh non-resume run.

Q15. Why do you remove old part files before a fresh run?
A15. To avoid mixing stale outputs with new outputs, which can cause duplication and inconsistent results.

Q16. Which feature engineering method is used for text?
A16. A saved vectorizer is used to transform clean_text into sparse numeric features; training and inference both use the same artifact.

Q17. Why do you save the vectorizer artifact?
A17. Saving the vectorizer guarantees feature-space consistency between training and production inference.

Q18. What extra numeric features were added?
A18. token_count, title_chars, abstract_chars, title_abstract_ratio, and log_token_count.

Q19. Why use log_token_count?
A19. log1p reduces skew in long-tail token distributions and can stabilize linear model behavior.

Q20. What supervised models did you train?
A20. Multinomial Naive Bayes, Logistic Regression, Linear SVM, Calibrated SVM, SGDClassifier, and Random Forest with SVD reduction.

Q21. Why include Multinomial Naive Bayes?
A21. It is a strong and fast baseline for sparse text classification and works well at scale.

Q22. Why include Logistic Regression?
A22. It is a reliable linear baseline for high-dimensional sparse text and often performs competitively.

Q23. Why include Linear SVM?
A23. Linear SVM is robust for text data and usually provides strong margins and good generalization.

Q24. Why include Calibrated SVM?
A24. Raw SVM scores are margins, not probabilities. Calibration converts them into more interpretable confidence-like probabilities.

Q25. Why include SGDClassifier?
A25. SGD supports incremental learning with partial_fit, making it practical for very large datasets.

Q26. Why combine Random Forest with SVD?
A26. Tree models on raw sparse TF-IDF-like features are expensive. SVD compresses dimensions first, making RF feasible.

Q27. Why did you use model-specific row caps?
A27. Some models are memory-heavy. Row caps prevent out-of-memory crashes while still allowing meaningful training.

Q28. Which metric did you use to select the best classifier?
A28. Macro F1 was prioritized to reduce bias toward majority classes.

Q29. Why not only accuracy?
A29. With class imbalance, accuracy can hide poor performance on minority categories. Macro F1 is more balanced.

Q30. What was your current best model artifact?
A30. The current saved report shows multinomial_nb as best among available scored models.

Q31. What is unsupervised clustering in your project?
A31. MiniBatchKMeans groups papers into coarse topic-like clusters without labels.

Q32. Why MiniBatchKMeans instead of full KMeans?
A32. MiniBatchKMeans scales better to large corpora by updating centroids from small batches.

Q33. What is topic modeling in your project?
A33. LDA (Latent Dirichlet Allocation) discovers latent word-topic structure across documents.

Q34. Why use online LDA?
A34. Online mode supports incremental partial_fit updates for large datasets and bounded memory usage.

Q35. What is the role of SentenceTransformer?
A35. It encodes text into dense semantic embeddings used for similarity search and retrieval.

Q36. Which embedding model did you use?
A36. sentence-transformers/all-MiniLM-L6-v2 for efficient and strong semantic retrieval.

Q37. Why normalize embedding vectors?
A37. L2 normalization allows inner product search to behave like cosine similarity.

Q38. Why use FAISS?
A38. FAISS provides fast nearest-neighbor search over dense vectors, ideal for semantic retrieval at scale.

Q39. What FAISS index type did you use?
A39. IndexFlatIP, which performs exact inner-product search.

Q40. What metadata do you keep with FAISS vectors?
A40. Paper id, title, abstract, and broad_category to map retrieved vector IDs back to readable paper information.

Q41. What is RAG in your system?
A41. Retrieval-Augmented Generation: first retrieve relevant papers/chunks, then generate answers grounded in that context.

Q42. Which generator models are used?
A42. flan-t5-base for RAG answering and flan-t5-small for paper chat, with optional cloud LLM backends.

Q43. Which summarization model is used locally?
A43. sshleifer/distilbart-cnn-12-6 is used for local summarization.

Q44. Why support cloud LLM mode?
A44. It reduces local compute and model download requirements, making deployment easier on low-resource machines.

Q45. Which API framework did you use?
A45. FastAPI, because it is fast, typed, and easy to build production-style ML endpoints.

Q46. What are key API endpoints?
A46. /classify, /search, /summarize, /ask, /agent/run, and chat endpoints for upload/load-arxiv/ask/session info.

Q47. How does /classify work internally?
A47. It preprocesses title+abstract, vectorizes with saved vectorizer, and predicts using the saved classifier artifact.

Q48. How does /search work internally?
A48. It embeds the query, performs FAISS nearest-neighbor search, then returns top_k matched papers from metadata.

Q49. How does /ask differ from /search?
A49. /search returns retrieved documents; /ask retrieves context and generates a natural-language answer.

Q50. What statistical tests did you run in EDA stage?
A50. Kruskal-Wallis, Chi-square test of independence, Cramer’s V effect size, bootstrap confidence interval, and permutation test.

Q51. Why use Kruskal-Wallis instead of ANOVA?
A51. Token count distributions are skewed and non-normal; Kruskal-Wallis is a non-parametric alternative.

Q52. Why compute Cramer’s V?
A52. Chi-square gives significance, while Cramer’s V gives effect size strength of association.

Q53. How do you ensure reproducibility?
A53. Fixed random_state values, deterministic hash splits, saved artifacts, and scripted pipeline stages.

Q54. What are the main artifact folders?
A54. classification, clustering, similarity, processed, feature_engineering, topic_modeling, and showcase.

Q55. What are current limitations?
A55. Class imbalance still affects minority-class recall, and some heavy models need row caps for memory safety.

Q56. What future improvements would you propose?
A56. Better imbalance handling (class weights/focal methods), hyperparameter search, advanced transformer fine-tuning, approximate FAISS indexes for larger scale, and stronger evaluation dashboards.
